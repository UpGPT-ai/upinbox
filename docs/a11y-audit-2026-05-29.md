# UpInbox Accessibility Audit — 2026-05-29

**Scope:** `src/components/` and `src/app/` (TSX surface area).
**Method:** Static grep audit for the six WCAG-adjacent failure modes
(missing alt text, unlabeled buttons, unlabeled links, click handlers on
non-interactive elements, unlabeled form inputs, color-only state
feedback). Each hit was hand-inspected to suppress false positives where
the element has visible text content or a wrapping `<label>`.

**Severity scale**
- **P0** — Blocks screen-reader users from completing a core flow, or
  produces silently broken assistive-tech behavior (unlabeled buttons,
  un-focusable click targets, form fields with no accessible name).
- **P1** — Degrades the experience (no announced state changes, tooltip
  used as primary label, color used as the only signal, links opened in
  new tab without warning).
- **P2** — Polish (decorative imagery without `aria-hidden`, redundant
  labels, focus-ring inconsistencies).

---

## P0 — Screen-Reader Blockers

### P0-1. Icon-only and emoji-only buttons rely on `title` instead of `aria-label`
`title` is **not** a reliable accessible name — most screen readers
ignore it on `<button>` when there is no other text. Multiple toolbar
controls in the mail UI fall into this trap.

| File | Lines | Control | Visible content |
|---|---|---|---|
| `src/components/mail/email-list.tsx` | 101 | Clear-search button | `✕` |
| `src/components/mail/email-list.tsx` | 308 | Master selection checkbox (rendered as `<button>`) | empty styled `<div>` |
| `src/components/mail/email-list.tsx` | 562, 570, 732, 741 | Snooze / Archive / Delete row actions | `🔔` / `📦` / `🗑` emoji only |
| `src/components/mail/email-list.tsx` | 411 | Generic `ToolbarAction` button (label rendered only ≥sm breakpoint) | icon + `<span className="hidden sm:inline">{label}</span>` |
| `src/components/mail/compose-window.tsx` | 326 | Remove-attachment button | `✕` |

The sort-direction toggle at line 111 (`↓ Date` / `↑ Date`) is fine —
it has text content.

**Fix:** add `aria-label` matching the existing `title` attribute on
every icon-only button. For the `ToolbarAction` component on line 411,
move `aria-label={label}` onto the `<button>` itself so the label is
announced at all viewport widths (not just ≥sm where the text span
becomes visible). For the master-selection control, add
`role="checkbox"` + `aria-checked={allSelected || partialSelected}` and
an `aria-label="Select all emails"`.

---

### P0-2. Click handlers on non-interactive `<div>` rows (no `role`, no keyboard handler)
Mail rows and bundle headers in the screener and smart-bundles views
fire `onClick` on a `<div>`. They are not focusable, do not respond to
Enter/Space, and have no implicit role — keyboard-only and SR users
cannot open an email from these surfaces.

| File | Lines | Element |
|---|---|---|
| `src/components/mail/smart-bundles.tsx` | 85 | Email row `<div onClick={() => onEmailClick(email.id)}>` |
| `src/components/mail/smart-bundles.tsx` | 134 | Bundle header `<div onClick={onToggle}>` |
| `src/components/screener/feed-email-list.tsx` | 145 | Email row `<div onClick={() => setOpenEmailId(email.id)}>` |
| `src/components/mail/unified-email-list.tsx` | 108 | Account row click |

**Fix (preferred):** convert each row to `<button type="button">` or
`<a>` with appropriate styling. **Fix (minimum):** add
`role="button"`, `tabIndex={0}`, and an `onKeyDown` handler that
triggers the same callback on `Enter` and `Space`, plus an
`aria-label` describing the row.

### P0-3. Form inputs with no accessible name
- `src/components/mail/unified-email-list.tsx:70` — global "Search…"
  input has only a placeholder. Placeholders are not accessible names.
- `src/components/mail/email-list.tsx:91` — main "Search emails…"
  input is keyboard-reachable but has no `<label>` and no
  `aria-label`. The adjacent `🔍` emoji is in a sibling `<span>`, not
  associated.

**Fix:** add `aria-label="Search emails"` (and `aria-label="Search all
inboxes"` for unified). For consistency, prefer a visually-hidden
`<label htmlFor>` over `aria-label`.

The advanced-filter inputs in `email-list.tsx` (lines 139, 149, 159,
168, 177) and `auto-archive-rules.tsx:368` are correctly wrapped in
`<label>` elements — **no fix needed** there.

## P1 — Degraded Experience

### P1-1. Status / error messaging is not announced
Several components show transient error or success text without
`role="status"`, `role="alert"`, or `aria-live`. SR users will not
hear them appear.

| File | Lines | Behavior |
|---|---|---|
| `src/components/screener/confidence-inspector.tsx` | 142 | "Something went wrong. Please try again." (`text-red-600`) |
| `src/components/mail/deep-clean-wizard.tsx` | 130, 324 | Wizard error banner |
| `src/components/mail/alias-manager.tsx` | 165 | Inline alias error |
| `src/components/mail/compose-window.tsx` | 378–381 | `✓ Sent` (`text-green-600`) and `⚠ {errorMsg}` (`text-red-600`) — send state |
| `src/components/settings/notification-rules.tsx` | 235, 334 | Validation error (red) and "Saved" toast (green, transitioned via opacity) |

**Fix:** wrap each in `<div role="status" aria-live="polite">` for
neutral/success and `role="alert"` (or `aria-live="assertive"`) for
errors. The compose-window send state is the most user-visible —
fixing it has the highest perceived impact.

---

### P1-2. State conveyed by color only
Even where text is present, the *category* of the message is signaled
exclusively by color (red = bad, green = good). Pair each with an icon
or text prefix so color-blind / monochrome users get the same signal.

| File | Lines | Pattern |
|---|---|---|
| `src/components/screener/feed-email-list.tsx` | 35, 38 | Tag pills `RECEIPT` (green) / `ACTION_REQUIRED` (red) — label is the tag name; add an icon prefix |
| `src/components/mail/alias-manager.tsx` | 264, 298 | Active/inactive alias state — bg/text color only |
| `src/components/mail/compose-window.tsx` | 378, 381 | "Sent" vs error distinguished by green/red plus `✓`/`⚠` (the icons help but the text token itself does not name the category) |

**Fix:** add a leading icon AND a text token (e.g. "Error: …",
"Success: …") on every color-coded status surface.

---

### P1-3. External links open in new tab without warning
`src/components/ai/ai-setup-panel.tsx:228, 237, 251` — three
`target="_blank"` links to `uplink.upgpt.ai` and `npmjs.com` without
any indication to AT users that focus will leave the page.

**Fix:** add `aria-label="Open UpLink documentation (opens in new
tab)"` (or equivalent) on each, and append a visible "↗" / "(new
tab)" hint.

---

### P1-4. Toggle / checkbox custom controls
`src/components/settings/notification-rules.tsx:113` — Toggle uses
`role="switch"` + `aria-checked` correctly (good), but lacks
`aria-labelledby`/`aria-label`. The surrounding row description is
not programmatically associated. Lines 282 and 297 contain bare
`<input>` elements — verify each has a wrapping `<label>` or
`htmlFor` (current grep flagged them; inspect with the surrounding
form rendering).

**Fix:** ensure each Toggle gets `aria-labelledby` pointing to the row
title element, or an explicit `aria-label`.

## P2 — Polish

### P2-1. Decorative imagery is already correctly hidden — keep doing this
`src/components/analytics/sender-view.tsx:96` — `<img alt=""
aria-hidden="true">` with a text-fallback `<div aria-hidden="true">`.
This is correct and is the only true `<img>` tag in the audited code.
The other grep hit (`src/components/mail/email-detail.tsx:66`) is a
regex string inside an HTML sanitizer, not a real `<img>`. **No fix
needed**, called out as a positive pattern to replicate.

### P2-2. Tooltip + emoji pattern is fragile
The row-action toolbar in `email-list.tsx` uses `title="Snooze"` on
emoji buttons. Even after the P0-1 `aria-label` fix, the emoji
character itself (`🔔`, `📦`, `🗑`) may be announced verbatim by some
screen readers ("bell", "package", "wastebasket"). Consider replacing
with inline SVG icons that are `aria-hidden="true"` so the
`aria-label` is the sole announced text.

### P2-3. `outline-none` without a visible focus replacement
Several inputs use `outline-none focus:ring-1 focus:ring-primary/50`
which is acceptable, but
`src/components/screener/confidence-inspector.tsx:146` uses
`focus:outline-none` with no compensating ring. Add a visible focus
indicator.

### P2-4. Buttons flagged by grep that are actually fine (no action)
These buttons have visible text content and just lack `aria-label`
(redundant in that case):
- `src/components/mail/snooze-selector.tsx` 168, 180, 203, 219
  ("Custom", "Set snooze", "Cancel", quick-option labels)
- `src/components/mail/follow-up-selector.tsx` 180, 193, 217, 238
- `src/components/screener/feed-tabs.tsx:65` (icon + visible label)
- `src/components/mail/smart-bundles.tsx:335`
- `src/components/settings/capability-badges.tsx:200` ("Manage
  subscription at UpGPT.ai")
- `src/components/screener/confidence-inspector.tsx` 126, 133, 145
  ("Correct", "This is correct", "Why was this categorized here?")
- `src/components/mail/search-suggestions.tsx:202` (already has
  `role="option"` and renders suggestion text)
- `src/components/ui/button.tsx:14` (base component — labelling is the
  caller's responsibility)

## Summary

| Severity | Distinct issues | Files touched |
|---|---|---|
| P0 | 3 (icon-only buttons; non-interactive clickable divs; unlabeled search inputs) | 5 |
| P1 | 4 (no live regions; color-only status; new-tab links; toggle labelling) | 6 |
| P2 | 4 (polish) | 5 |

**Suggested order of work**
1. **P0-1** (`aria-label` sweep on icon-only buttons) — cheapest,
   biggest SR impact, can ship in one PR.
2. **P0-2** (convert clickable rows to `<button>`) — required for
   keyboard-only navigation of the inbox.
3. **P0-3** (search input labels).
4. **P1-1** (`role="status"` / `aria-live` on send / error / save
   surfaces) — composer is the highest-traffic surface.
5. Remaining P1/P2.

**Out of scope of this static audit**
- Color-contrast ratios (requires rendered DOM + token resolution).
- Focus-trap behavior in modals / popovers (`SnoozeSelector`,
  `FollowUpSelector`, `ComposeWindow`).
- Keyboard shortcut conflicts with screen-reader virtual cursor.
- Reduced-motion preferences on animated `transition-*` classes.

Recommended follow-up: run `axe-core` against the running app in
Playwright (project already has Playwright config) to cover the
dynamic surface area this static audit cannot reach.
